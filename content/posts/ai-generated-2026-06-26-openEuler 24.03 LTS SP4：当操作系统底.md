传统操作系统内存管理的核心假设是"静态容量规划"，而AI工作负载的显著特征恰恰是**内存需求的剧烈波动**——训练时的峰值吞噬、推理时的稀疏访问、模型切换时的冷热迁移。

### 64K 内核页：为海量数据流重构地址翻译

Linux 默认的 4KB 页表设计源于上世纪90年代的内存容量约束。当AI训练面临**TB级参数模型**和**GB级批量数据**时，页表遍历的TLB压力成为squeeze性能的关键瓶颈。

SP4 的 64K 内核选项并非简单放大：

| 维度 | 4K 页表 | 64K 页表 | 收益场景 |
|:---|:---|:---|:---|
| TLB 覆盖率 | 256项 × 4KB = 1MB | 256项 × 64K = 16MB | 大模型参数热区访问 |
| 页表深度 | 5级 (x86_64) | 3级优化 | 减少内存访问次数 |
| 缺页中断频率 | 高 (16×) | 低 | 降低内核态切换开销 |
| 内存碎片 | 细粒度 | 需配合大页预分配 | 连续内存分配优化 |

关键挑战在于**64K内核与4K用户态的兼容性**。SP4 的解决方案是**混合页表映射**：内核空间采用64K粒度以优化I/O和内核数据路径，用户空间通过`madvise(MADV_HUGEPAGE)`按需提升大页，同时保留`CONFIG_COMPAT_4K`选项确保遗留应用兼容。

```c
// SP4 内核中 64K 页配置的关键路径示意
// arch/x86/mm/init.c 简化逻辑

#ifdef CONFIG_OE_64K_PAGES
void __init oe_configure_page_size(void)
{
    /*
     * 检测平台能力：CPU是否支持 64K 子页划分 (Intel TME-MK / AMD SME)
     * 若支持，启用透明 64K 提升；否则回退到显式 hugetlbfs
     */
    if (cpu_feature_enabled(X86_FEATURE_PKS) || 
        boot_cpu_has(X86_FEATURE_SME)) {
        set_page_size_shift(PAGE_SHIFT_64K);
        /* 关键：保持 PTE 结构兼容，通过 software-defined bit 标记子页状态 */
        enable_subpage_tracking();
    } else {
        pr_info("openEuler: 64K pages require TME-MK/SME, falling back\n");
    }
}
#endif
```

### AI 图编译器：操作系统介入"编译-执行"边界

这是 SP4 最具野心的架构创新。传统认知中，操作系统位于"编译器之下、硬件之上"；而 SP4 的**AI图编译器集成**打破了这层边界。

以**算子融合**为例：静态编译器基于启发式规则决定哪些算子融合，但无法预知运行时的**实际张量形状**（dynamic shape）和**内存压力**。SP4 的内置分析器在首次warmup迭代中收集真实执行特征，反馈给图编译器进行**迭代优化**——这类似于JVM的JIT编译，但发生在更底层的系统层面。
