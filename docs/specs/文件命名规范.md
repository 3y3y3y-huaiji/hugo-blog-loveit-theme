# 文件命名规范

## 1. 通用命名原则

1. **清晰性**：文件名应清晰反映文件内容
2. **一致性**：在整个项目中保持一致的命名风格
3. **可读性**：使用有意义的名称，避免缩写（除非是广泛认可的缩写）
4. **简洁性**：保持文件名简洁，但不要牺牲清晰性
5. **无特殊字符**：避免使用空格和特殊字符（如 `!@#$%^&*()`）

## 2. 命名格式规范

### 2.1 文件名格式

- **小写字母**：文件名应全部使用小写字母
- **单词分隔**：单词之间使用短横线 `-` 分隔（kebab-case）
- **文件扩展名**：使用适当的文件扩展名

**示例**：
```
blog-repository-optimization.md  # 正确（使用短横线分隔单词，小写）
BlogRepositoryOptimization.md    # 错误（使用驼峰命名）
blog_repository_optimization.md  # 错误（使用下划线分隔）
```

### 2.2 目录名格式

- **小写字母**：目录名应全部使用小写字母
- **单词分隔**：单词之间使用短横线 `-` 分隔（kebab-case）

**示例**：
```
content/posts/               # 正确
content/Posts/               # 错误（大写字母）
content/blog_posts/          # 错误（使用下划线）
```

## 3. 不同类型文件的命名规范

### 3.1 Markdown 内容文件

- **文章文件**：使用描述性标题，单词之间用短横线分隔
  - 示例：`effective-note-taking.md`
- **页面文件**：使用页面功能命名
  - 示例：`about/index.md`（关于页面）
- **索引文件**：使用 `_index.md`（Hugo 要求）

### 3.2 配置文件

- **主配置文件**：使用 `hugo.toml`（Hugo 3.0+ 推荐）
- **主题配置**：使用主题提供的配置文件（通常在 `themes/[theme-name]/` 目录）
- **数据文件**：使用描述性名称，单词之间用短横线分隔
  - 示例：`site-settings.yaml`

### 3.3 代码文件

- **HTML 模板**：使用描述性名称，反映模板功能
  - 示例：`single.html`（单页模板）、`list.html`（列表模板）
- **CSS/SCSS 文件**：使用描述性名称，反映样式功能
  - 示例：`main.scss`（主样式文件）、`blog-post.scss`（博客文章样式）
- **JavaScript 文件**：使用描述性名称，反映脚本功能
  - 示例：`theme.js`（主题功能脚本）、`search.js`（搜索功能脚本）

### 3.4 静态资源文件

- **图片文件**：使用描述性名称，反映图片内容
  - 示例：`avatar.png`（头像）、`blog-banner.jpg`（博客横幅）
- **图标文件**：使用图标名称或功能命名
  - 示例：`favicon.ico`、`logo.png`
- **第三方库文件**：保持原始文件名，但确保版本清晰
  - 示例：`jquery-3.6.0.min.js`

### 3.5 文档文件

- **项目文档**：使用中文名称，清晰反映文档内容
  - 示例：`需求文档.md`、`项目状态.md`
- **技术文档**：使用描述性名称，反映文档主题
  - 示例：`目录结构规范.md`、`文件命名规范.md`

## 4. 特殊文件命名

### 4.1 国际化文件

- 使用语言代码命名
  - 示例：`zh-cn.toml`（中文）、`en.toml`（英文）

### 4.2 原型模板文件

- 使用内容类型命名
  - 示例：`post.md`（文章原型）、`page.md`（页面原型）

- [ ] 文件名是否全部使用小写字母？

- [ ] 文件名是否清晰反映文件内容？
- [ ] 是否避免使用特殊字符和空格？
- [ ] 是否保持与项目中其他文件命名风格一致？
- [ ] 配置文件和特殊文件是否使用了正确的名称？
